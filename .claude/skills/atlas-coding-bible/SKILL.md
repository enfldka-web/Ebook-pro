# Atlas Coding Bible

## 목적
Atlas는 수개월간 이어지는 장기 프로젝트다. 세션이 바뀌어도, 누가 작업하든
항상 같은 방식으로 안전하게 개발되어야 한다. 이 스킬은 그 개발 방식을 고정한다.

## 관련 스킬
- **`atlas-business-bible`** — 왜 이 작업을 하는지(최상위 철학, 가장 먼저 참조)
- **`atlas-product-bible`** — 애초에 이 기능/방향을 만들어도 되는지
- **`atlas-design-bible`** — 디자인/UI/Brand Theme가 조금이라도 관련된 작업이면
  반드시 함께 사용한다. 이 스킬은 "어떻게 안전하게 개발할지"를 다루고,
  `atlas-design-bible`은 "무엇을 만들어야 하는지(디자인 규칙)"를 다룬다.
- **`atlas-qa-bible`** — 개발이 끝난 뒤 실제로 통과했는지 확인하는 방법과
  보고 형식

다섯 스킬은 서로 대체하지 않는다. business-bible을 가장 먼저 참조하고, 작업에
여러 역할이 겹치면(예: 디자인이 포함된 새 기능) 관련 스킬을 함께 사용한다.

## 문서/지시 우선순위
서로 다른 지시나 문서가 충돌하는 것처럼 보일 때는 아래 순서로 우선한다.
1. 사용자의 현재 명시적 지시
2. 프로젝트 Skill — 이 안에서도 `atlas-business-bible`(왜) →
   `atlas-product-bible`(무엇을) → `atlas-design-bible` → `atlas-coding-bible`
   → `atlas-qa-bible` 순으로 참조한다
3. `design.md` 및 `docs/` 문서
4. 기존 구현 관례

다만 이 순서는 "임의로 판단해도 된다"는 뜻이 아니다. 문서 간 충돌이 실제로
발견되면 이 순서를 참고만 하고, 어느 쪽을 따를지는 임의로 선택하지 말고
반드시 사용자에게 먼저 질문한다(Never Guess와 동일한 태도). 이 정합성 점검
이후로는 `design.md`와 `atlas-design-bible`이 같은 기준을 갖도록 유지한다 —
둘이 다시 어긋나면 그 자체가 먼저 사용자에게 확인해야 할 충돌이다.

## 핵심 행동 원칙

### Never Guess
추측하지 않는다. 판단이 애매하거나, 이 스킬/`design.md`/기존 코드 어디에도
답이 없거나, 사용자의 기존 지시와 충돌할 가능성이 있으면 **절대 임의로
결정하지 않는다.** 반드시 먼저 사용자에게 질문한다. "이렇게 하는 게 나아
보여서"라는 이유로 요청받지 않은 구조/데이터/동작을 만들어내지 않는다.

### UX First
기능 개수보다 사용 경험을 우선한다. 무언가를 더 추가할지 고민될 때는 먼저
"기존 기능을 더 쉽고 직관적으로 만들 방법은 없는가"부터 검토한다.

### Ship-Quality Bar
완성된 구현이 실제로 출시 가능한 품질인지 판단하는 개발 품질 기준이다 (어떤
기능을 만들지 판단하는 `atlas-product-bible`의 "Commercial First"와는 다른
질문 — Commercial First는 "무엇을 만들까", Ship-Quality Bar는 "다 만든 게
정말 끝난 건가"를 묻는다). 임시 구현을 최종 결과로 간주하지 않는다. Placeholder를
최종 결과로 간주하지 않는다. Demo UI를 최종 결과로 간주하지 않는다. TODO가
남은 상태를 완료로 보고하지 않는다. 사용자에게 "완료"라고 보고하기 전에, 실제
유료 SaaS에 배포 가능한 수준인지 스스로 점검한다.

### Bug Discovery
작업 중 버그를 발견하면 그냥 넘어가거나 별도 보고만 하고 방치하지 않는다.
**즉시 수정**하고, 사용자에게 (1) 원인, (2) 어떻게 고쳤는지, (3) 재검증 결과를
함께 보고한다.

### 대안 제시 원칙
사용자가 "더 좋은 방법이 없을까?"라고 물으면 현재 구현을 방어하는 데 그치지
않고, 실제로 고려할 만한 대안을 함께 제안한다. 다만 **최종 결정은 항상
사용자에게 맡긴다** — 대안을 제시하되 임의로 갈아엎지 않는다.

## 핵심 불변식 (절대 위반 금지)
- **기존 기능 삭제 금지** — 요청받지 않은 기능/화면/버튼을 제거하지 않는다.
- **기존 API 변경 금지** — 기존 공개 함수명, 이벤트 연결 계약, 외부에서
  참조되는 인터페이스를 사용자 승인 없이 변경하지 않는다.
- **기존 데이터 구조 변경 금지** — 기존 저장·불러오기 계약과 데이터 호환성을
  깨지 않는다. 필드를 추가할 때도 기존 필드는 그대로 두고, 레거시 프로젝트(새
  필드가 없는 저장본)를 불러와도 에러 없이 동작해야 한다.
- **Preview와 Export 결과는 항상 동일해야 한다** — 화면에 보이는 것과 실제
  다운로드되는 PNG/JPG/ZIP이 달라지는 변경은 그 자체로 실패다.

## Git 워크플로우
1. 작업을 시작하기 전 **항상 최신 main과 동기화됐는지 확인**한다
   (`git fetch`/`git status`로 뒤처짐 여부 확인).
2. 기능/수정 단위마다 **항상 새 Branch를 만들고 그 위에서 작업**한다. main에
   직접 커밋하지 않는다.
3. **Commit은 사용자 승인 후에만** 수행한다. 승인 없이 먼저 커밋하지 않는다.
4. **Push는 사용자 승인 후에만** 수행한다. 승인 없이 원격에 올리지 않는다.
5. 사용자가 "커밋하지 마세요"/"승인 전"이라고 명시한 동안에는, 자동 Stop
   hook이 "uncommitted changes" 경고를 반복해도 그 경고를 이유로 임의 커밋하지
   않는다 — 사용자의 명시적 지시가 자동 훅 메시지보다 항상 우선한다.

## Phase 단위 개발
큰 기능은 한 번에 끝내지 않고 **Phase 단위로 나눠서** 진행한다(예: 상태 설계 →
기본 UI → 데이터 연동 → Export → QA). 각 Phase는 독립적으로 완료·검증 가능한
단위여야 한다.

## Phase 종료 시 검증 (건너뛰지 않음)
각 Phase가 끝나면 다음을 실제로 실행해 확인한다:
1. **Playwright 검증** — 실제 브라우저에서 해당 Phase가 구현한 화면/동작을
   클릭·입력해 눈으로/스크린샷으로 확인한다. 코드만 읽고 "동작할 것"이라고
   추정하지 않는다.
2. **Regression Test** — 이번 변경이 기존 기능(로그인, 대시보드, 전자책 생성
   플로우, Thumbnail Studio, Sales Page Studio, 저장/불러오기 등)을 깨지
   않았는지 확인한다. 특히 상태를 공유하는 기능(예: Brand Theme 공유 상태)은
   두 화면 모두에서 확인한다.
3. **Preview == Export 동일성 확인** — 화면 변경이 있었다면, 실제로 Export를
   실행해 파일 크기/색상이 Preview와 일치하는지 (가능하면 픽셀 치수까지)
   교차 검증한다.
4. **기존 기능 회귀 여부 확인** — 위 1~3에서 하나라도 실패하면 다음 Phase로
   넘어가지 않고 먼저 고친다.

## 문서 업데이트 규칙
작업 성격에 따라 `docs/` 아래 관련 문서를 갱신한다: 기능을 추가/변경/완료했으면
변경 이력·현재 기능 목록을 다루는 문서를, 버그를 발견했거나 고쳤으면 버그 추적
문서를, Milestone 단위 작업이 끝났으면 마일스톤 추적 문서(필요 시 별도 완료
보고서)를 갱신한다. 현재 어떤 문서가 어떤 역할을 하는지는 `docs/ARCHITECTURE.md`
와 `docs/` 폴더 자체를 확인한다 — 구체적인 파일명은 프로젝트 구조가 바뀌면 함께
바뀔 수 있으므로 이 스킬에 고정하지 않는다. 문서 갱신도 코드와 마찬가지로
사용자 승인 흐름을 따른다 — 커밋 시점은 코드와 동일하게 사용자 승인 후.

## 최소 수정 원칙
요청받은 것보다 더 많이 고치지 않는다. 리팩토링, 이름 변경, "더 나아 보이는"
구조 변경을 요청받지 않았다면 하지 않는다. 필요한 코드만 최소한으로 수정하고,
관련 없는 파일/라인은 건드리지 않는다.
