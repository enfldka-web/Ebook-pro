# Atlas QA Bible

## 목적
`atlas-coding-bible`은 "Phase가 끝나면 검증한다"는 원칙을 말한다. 이 스킬은
그 검증을 **무엇을, 어떻게, 어떤 순서로, 어떤 형식으로 보고할지** 구체적으로
정의한 실행 체크리스트다. UI나 기능 개발을 마쳤다면, 사용자가 요청하지 않아도
"완료"라고 보고하기 전에 항상 이 체크리스트를 거친다. 이 스킬에는 변하지 않는
QA 원칙과 검증 절차만 담는다 — 현재 구현 상태, 특정 마일스톤, 특정 버그 번호,
현재 파일 목록처럼 계속 바뀌는 정보는 `docs/` 문서와 각 작업의 완료 보고서가
관리한다.

## 디자인 점검
- Preview가 화면의 중심인가 (`atlas-design-bible`의 Preview First 원칙)
- Canva, Gamma, Framer, Apple의 UX 철학에 부합하는가 — 개발자 도구처럼 보이지 않는가
- Premium / Study Note / Handwriting의 차이가 색상만 다른 스킨이 아니라
  명확하게 다른 브랜드 세계로 느껴지는가
- Brand Pack이 화면 전체(카드/버튼/뱃지/체크리스트/CTA/Export)에 일관성 있게
  적용되는가
- 여백과 계층 구조가 적절한가 — 무엇이 가장 중요한 요소인지 한눈에 보이는가
- 선택 상태와 Active 표시가 실제 화면에서 일치하는가
- 밝은 테마에 다크 컨트롤이 남아 있지 않은가 (하드코딩된 어두운 색이 밝은
  테마에도 그대로 남아있지 않은지)
- 입력 중 포커스와 커서 위치가 유지되는가 (재렌더링 때문에 타이핑 중 포커스가
  끊기지 않는지)
- Preview가 설정 패널보다 시각적으로 우선되는가
- 작은 화면에서 레이아웃이 깨지지 않는가
- 편집용 테두리, 선택 표시, 버튼이 Export에 포함되지 않는가
- 한글 글꼴이 깨지거나 Fallback으로 인해 의도와 다르게 보이지 않는가

스크린샷을 생성했다면 자동 테스트 결과만 보지 말고 반드시 육안으로도
확인한다.

## 기능 점검
- Preview와 Export가 동일한가 (색상/치수 모두)
- 저장과 불러오기가 정상인가
- 기존 기능이 깨지지 않았는가
- 기존 프로젝트(새 필드가 없는 이전 저장본 포함)와 하위 호환되는가

## Export 검증

**Thumbnail**
- 실제 크기 652×488px
- 4:3 비율 유지
- PNG/JPG 파일이 실제로 열리는지 확인

**Sales Page**
- 기준 DOM 540×675px
- 실제 PNG 1080×1350px
- ZIP 내부 파일 수가 활성 섹션 수와 일치하는지 확인
- ZIP 내부 파일명과 순서가 현재 화면 순서와 일치하는지 확인

이 규격은 현재 승인된 Atlas 제품 사양이다. 향후 사용자가 규격 변경 또는 다중
규격 지원을 승인하면 관련 Skill과 `design.md`를 함께 갱신해야 한다.

**공통**
- Preview와 실제 파일의 색상과 레이아웃이 일치하는가
- 실패한 파일이 있는데 전체 성공으로 보고하지 않는다
- 중복 클릭 방지가 되어 있는가
- 라이브러리 미로드 시 명확한 오류가 표시되는가

## 테스트 절차
1. **Playwright 테스트 수행** — 실제 브라우저에서 클릭/입력해 동작을 확인한다.
2. **Regression Test 수행** — 이번 변경과 무관해 보이는 기존 기능까지 함께
   확인한다.
3. **Console Error 확인** — `pageerror`/`console.error`가 새로 발생하지
   않는지 확인한다.
4. **Export 규격 확인** — 위 "Export 검증" 섹션 기준으로 실제 파일을 검사한다.
5. **PNG/JPG/ZIP 결과 확인** — 실제로 파일을 열어(또는 동등하게 검증해) 테마별
   색상/구성이 Preview와 일치하는지 확인한다.

## 실제 검증과 대체 검증 구분
실제 라이브러리, 실제 API, 실제 네트워크 환경을 사용하지 못한 경우에는 그
테스트를 실제 검증으로 표현하지 않는다. Stub, Mock, Fallback, 합성 데이터,
DOM 직접 주입, 브라우저 캐시, 대체 라이브러리 등을 사용했다면 반드시 **대체
검증**이라고 명시한다.

예시:
- 실제 Claude API 호출 미검증
- html2canvas Stub으로 파이프라인만 검증
- JSZip 대체 검증
- 합성 전자책 데이터로 UI만 검증

검증하지 못한 항목은 완료 보고서의 "실제 미검증 항목"/"남은 제한사항"에
기록한다.

## QA 통과 기준
다음 항목 중 하나라도 실패하면 작업을 완료로 보고하지 않는다.
- 주요 기능 테스트 실패
- 기존 기능 Regression 발생
- Preview와 Export 불일치
- 저장/불러오기 오류
- 하위 호환성 오류
- ReferenceError
- TypeError
- SyntaxError
- 신규 Console Error
- Export 규격 불일치
- 사용자에게 숨긴 제한사항 존재

실패 항목을 먼저 수정하고 재검증한 뒤에만 완료로 보고한다.

## 버그 보고 규칙
개발 또는 QA 중 버그를 발견하면 숨기지 않는다. 각 버그에 대해 반드시 다음을
기록한다.
- 증상
- 근본 원인
- 수정 내용
- 재검증 결과
- 기존 기능 영향 여부

테스트 스크립트 자체의 오류와 실제 앱 버그를 구분해서 보고한다.

## Never Hide
테스트하지 못한 기능, 실패한 항목, 환경 제약, 대체 검증, 남은 위험 요소를
숨기지 않는다. 사용자에게 정확하게 보고한다. **테스트하지 않은 기능을
통과했다고 표현하지 않는다.**

## 테스트 산출물 관리
Playwright 스크립트, 스크린샷, 임시 HTML, 로그, Export 결과물, ZIP, Patch,
백업 파일 등은 기본적으로 저장소 밖 임시 폴더에 생성한다. 사용자가 명시적으로
요청하지 않은 테스트 산출물은 Git 저장소에 포함하지 않는다. Commit 전에
반드시 저장소 오염 여부를 확인한다.

## 완료 보고 형식
QA를 마치면 항상 아래 17개 항목으로 보고한다:
1. 테스트한 환경
2. 변경 파일 목록
3. Playwright 결과
4. Regression Test 결과
5. Preview와 Export 동일성 결과
6. 실제 Export 규격
7. 저장/불러오기 결과
8. 하위 호환성 결과
9. 발견한 버그
10. 수정한 버그
11. 실제 미검증 항목
12. 남은 제한사항
13. Console Error
14. 저장소 오염 여부
15. git status
16. Commit 여부
17. Push 여부

## 관련 스킬
- **`atlas-business-bible`** — 왜 이 기준으로 통과/실패를 판단하는지(최상위 철학)
- **`atlas-product-bible`** — 이 기능/방향이 맞는지
- **`atlas-design-bible`** — 무엇을 시각적으로 만들지
- **`atlas-coding-bible`** — 어떻게 안전하게 개발할지
- **`atlas-qa-bible`(이 스킬)** — 다 만든 뒤 실제로 통과했는지 확인하는 방법과
  보고 형식

다섯 스킬은 business-bible(왜) → 제품 방향 → 디자인 규칙 → 개발 프로세스 →
품질 검증 순서로 함께 적용한다.
